# S-10 タスクフォームモーダル 方針

## 構成

- **背景**: タスクの新規作成・編集を行うモーダル。S-04 案件詳細のタスクタブから「タスク追加」ボタン or 行クリックで開く
- **レイアウト**: 画面中央に配置するモーダル、max-w-2xl（約 640px）、背景 dimmed（`bg-black/50`）
- **パンくず**: なし（モーダル）
- **URL**: モーダル表示中は URL は変わらない。ディープリンクが必要になれば `#task/{task_id}` でフラグメント対応
- **コンポーネント**: `TaskFormDialog`（shadcn/ui Dialog ベース）

## モーダル構成（上から下へ）

### ヘッダー
- 左: アイコン（list-checks、muted背景の円）
- タイトル:
  - 新規作成: 「タスクを追加」
  - 編集: 「タスクを編集」 + `TASK-042-005` のような task_id（mono、muted）
- 右: × ボタン（閉じる）

### フォーム本文

#### 1. タイトル *（必須）
- テキスト入力、maxlength=120、placeholder「例：IoTゲートウェイ設計書作成」
- 右下に文字数カウンター（`0 / 120`、mono）

#### 2. 種類チップ選択 *（必須、4択）
- `タスク` / `機能追加` / `改善` / `バグ`
- チップUI: 丸角 pill、選択時は背景色が種類別（task=灰 / feature=青 / improve=緑 / bug=赤）
- 非選択時は枠線のみ
- 選択チップのクリックで切替（ラジオ的挙動、1つだけ選択）

#### 3. 優先度チップ選択 *（必須、3択）
- `高`（赤） / `中`（黄） / `低`（青）
- 同じチップUI、選択状態で背景色を変える

#### 4. ステータスチップ選択 *（必須、3択）
- `未着手`（灰） / `進行中`（青） / `完了`（緑）
- ステータス変更で後続フィールドの挙動が変わる（条件付きレンダー）

#### 5. 進捗スライダー（条件付き）
- **ステータス=進行中 のときのみ表示**
- スライダー（0-100%、刻み5%）+ 右側に大きなパーセンテージ表示
- 未着手時は自動で 0%、完了時は自動で 100% に設定（バックエンド側）
- 進行中だけがユーザー編集可能

#### 6. 担当者セレクト
- ドロップダウン、選択肢に avatar + 氏名
- 選択肢は `projects.department_id` に所属するユーザーに絞る（他部門は表示しない）
- 「未割当」オプションを含む

#### 7. 期日
- date input（YYYY-MM-DD）
- 入力後に「あと ○ 日」or「○ 日超過」を右側に表示
- 超過時は赤文字

#### 8. 説明
- textarea、rows=4、placeholder「タスクの詳細や受入基準を記入」

#### 9. コメント
- **既存コメント表示**: タイムライン風（avatar + 氏名 + 日時 + 本文）
- **新規コメント入力**: textarea + 「コメント投稿」ボタン（右下）
- コメントは `task_comments` テーブルに保存

#### 10. 変更履歴（collapsible）
- 閉じた状態: 「変更履歴を表示（3件）」+ chevron-down アイコン
- 開いた状態: タイムライン形式で過去の変更（誰が / いつ / 何を変えたか）
- `task_histories` テーブルから `TaskHistoryService` が自動記録

### フッター
- **左**: 「削除」ボタン（btn-outline-red、trash アイコン）— 編集時のみ表示、新規作成時は非表示
- **右**: 「キャンセル」（btn-outline）+ 「保存」（btn-primary）
- 削除と保存を左右に分離することで誤操作を防ぐ

## ロール別アクセス

| ロール | タスク CRUD |
|---|---|
| applicant | 自案件（主担当）のみ○ |
| dept_manager | 自部門のタスクを閲覧・編集○ |
| hq_manager | 全タスクを閲覧・編集○ |

## バリデーション

- タイトル: 必須、1-120文字
- 種類・優先度・ステータス: 必須
- 期日: 任意。入力時は過去日も許容（遅延タスクを追加する運用のため）
- 進捗率: ステータス=進行中のときのみ 0-100 の数値

## 設計上のこだわり

- **チップ選択 UI を多用** — 種類・優先度・ステータスは select ではなくチップ式。select より「選択肢が見えている状態」で、タップ数が減り（モバイルで1タップ）、色分けで視覚的に即座に判別できる。Backlog や Linear 系のタスク管理ツールで主流の UI パターン
- **ステータス=進行中でのみ進捗スライダーを表示** — 未着手は必ず 0%、完了は必ず 100%。進行中だけが可変という状態機械の制約を UI で体現。「未着手なのに進捗 50%」のような矛盾状態を入力段階で不可能にする
- **削除ボタンを左、保存を右** — 破壊的アクション（削除）と主要アクション（保存）を物理的に離す。右手で保存を押す動線上に削除を置かないことで、誤タップのリスクを下げる。削除には確認ダイアログを別途挟む
- **変更履歴を collapsible に** — フォームの本題（入力項目）の邪魔にならない位置に折り畳み配置。監査証跡としての価値はあるが、毎回開く必要はないため、必要な時だけ展開
- **コメントをソーシャル風タイムライン** — タスクは担当者間のコミュニケーション場にもなる。Slack や Jira 的なコメントフィードを再現し、作業履歴が流れで読めるようにする
- **担当者候補は部門内に絞る** — 全社員から選ぶと選択肢が膨大。`projects.department_id` で絞ることで、選択ミスと認知負荷を両方下げる。他部門へのアサインが必要になれば課題2で拡張
- **task_id を編集時のみ表示** — 新規作成時は未採番なので出さない。編集時は `TASK-042-005` のように PRJ-ID を継承した識別子で、案件とタスクの親子関係がIDだけで分かるようにする
- **モーダルの max-w-2xl（640px）** — フォーム項目が多い（10項目）のでやや広め。狭すぎるとコメントが読みづらく、広すぎると入力フィールドが横伸びして読みにくい
