# シーダー シナリオパターン（案件・タスク・通知）

> 本ドキュメントは **`ProjectSeeder` / `DemoWorkloadSeeder` で再構築するデモデータのシナリオ一覧** です。
> **対象**: 案件 (`projects` + `approvals`)、タスク (`tasks`)、通知 (`notifications`)。
> **対象外（変更しない）**: `UserSeeder` / `DepartmentSeeder` / `RolePermissionSeeder`。
> **正本との関係**: スコープ・状態遷移・通知タイプの正本は `materials/Design/system_spec.md` と `materials/Design/er_diagram.md`。本ファイルはそれらを満たすデータ網羅のための **シナリオ列挙**。

## 3. タスク（tasks）シナリオパターン

### 割り当て要件

- **承認済みのすべての案件（10件）に対し、すべてのユーザー（10名）を担当者または確認者として割り当てる**。
- 各承認済案件 1 件につき、ステータス・優先度・期限のバリエーションを変えて **14〜16 タスク** を生成。
- `assignee_id` は 10 名を巡回して当て、`reviewer_id` は同部門の部門管理者を基本に、一部のタスクで別ユーザーをアサイン。
- 全タスクが「**ステータス4値 × 期限の状態（過去/直近/将来/null）**」の組み合わせを 2 件以上ずつ含む。

### シナリオパターン（各承認済案件で網羅）

| # | シナリオ | `status` | `progress_rate` | `due_date` | 通知発火 |
|---|---------|---------|-----------------|------------|----------|
| T-01 | 未着手・将来期限 | `open` | 0 | 今日+14日〜+30日 | （なし） |
| T-02 | 未着手・期限間近（3日以内） | `open` | 0 | 今日+1〜+3日 | `task_due_soon` |
| T-03 | 未着手・期限超過 | `open` | 0 | 今日-5〜-1日 | `task_due_soon`（過去） |
| T-04 | 未着手・期限未設定 | `open` | 0 | null | （なし） |
| T-05 | 進行中・初期段階 | `in_progress` | 15〜30 | 今日+5〜+10日 | `task_assigned` |
| T-06 | 進行中・終盤 | `in_progress` | 60〜85 | 今日+1〜+3日 | `task_due_soon` |
| T-07 | 進行中・期限超過 | `in_progress` | 40〜60 | 今日-3〜-1日 | `task_due_soon` |
| T-08 | 確認待ち（resolved・実装者の完了報告後） | `resolved` | 95〜100 | 今日-1〜+2日 | `task_resolved` |
| T-09 | 完了（closed・確認OK済） | `closed` | 100 | 今日-7〜+5日 | `task_reviewed` / `task_completed` |
| T-10 | 担当者未割当（バックログ） | `open` | 0 | null | （なし） |
| T-11 | 本部承認時自動投入「実装計画作成」（残置） | `open`〜`closed` | 0〜100 | 承認+10日 | `task_assigned`（既存） |
| T-12 | バグ・高優先度 | `open` / `in_progress` | 0〜50 | 今日+1〜+5日 | — |
| T-13 | 改善・低優先度 | `open` | 0 | 今日+30日〜 | — |
| T-14 | 機能追加・中優先度・確認者別ユーザー | `in_progress` / `resolved` | 50〜95 | 今日+5〜+10日 | `task_resolved` |

> 各シナリオを **同一案件内で A/B 2件ずつ** 生成。`assignee_id` を順に巡回させ、**全ユーザー 10 名が、いずれかの承認済案件で「担当者として最低 1 件、確認者として最低 1 件」を持つ**ことを保証する。

### 集計

- **承認済案件**: 10 件
- **1 案件あたりのタスク数**: 14〜16 件
- **タスク合計（目安）**: **140〜160 件**

### 補足

- `task_type`: `bug` / `feature` / `improvement` / `task` を均等に分散
- `priority`: `high` / `medium` / `low` を均等に分散
- `category`: `null`（課題1 では未使用）
- `parent_id` / `milestone_id`: `null`（将来用 FK）
- `created_by`: 申請者または該当部門の部門管理者
- `actual_days`: ステータスに応じた係数（`closed`=0.92、`resolved`=0.55、`in_progress`=0.38、`open`=0）

## 4. 通知（notifications）シナリオパターン

### 9 タイプ × 未読 / 既読の 2 パターン

| # | シナリオ | `type` | 受信者バリエーション | `read_at` |
|---|---------|--------|----------------------|-----------|
| N-01 | 申請提出（受領通知） | `project_submitted` | A=申請者本人 / B=部門管理者 | A=null / B=既読 |
| N-02 | 承認完了（本部承認後） | `project_approved` | A=申請者 / B=主担当 | A=null / B=既読 |
| N-03 | 却下（コメント付） | `project_rejected` | A=申請者(部門却下) / B=申請者(本部却下) | A=null / B=既読 |
| N-04 | 取り戻し（本部直行案件） | `project_returned` | A=部門管理者 / B=本部管理者 | A=null / B=既読 |
| N-05 | タスク担当割当 | `task_assigned` | A=新担当者 / B=別ユーザー | A=null / B=既読 |
| N-06 | タスク期限間近 | `task_due_soon` | A=担当者(期限3日前) / B=担当者(当日) | A=null / B=既読 |
| N-07 | 確認依頼（resolved） | `task_resolved` | A=確認者 / B=主担当 | A=null / B=既読 |
| N-08 | 確認OK | `task_reviewed` | A=実装者 / B=申請者 | A=null / B=既読 |
| N-09 | タスク完了（互換） | `task_completed` | A=申請者 / B=主担当 | A=null / B=既読 |

### 全ユーザー網羅ルール

- 上記 9 シナリオを **8 名（実体 10 名）全員に最低 1 件ずつ** 配るように受信者を巡回。
- 各ユーザーに **未読 2〜3 件・既読 1〜2 件** を持たせ、ヘッダーの未読バッジ・通知一覧の動作確認に十分な量を生成。

### 集計

- **9 種 × 2 パターン × 受信者バリエーション** = **約 70〜90 件**

### 通知ペイロードの規約

- `title`: NotificationType ごとに固定文（例: 「タスクの確認依頼があります」「本部承認が完了しました」）
- `body`: 案件タイトルやタスク名を含む 1 文
- `meta`: `{ "project_id": <id>, "task_id": <id?> }`
- `read_at`: 未読 = `null`、既読 = `now - random(1〜10日)`
- `is_read`: `read_at` に追従

## 5. 削除/再生成スコープ

| 対象 | 操作 |
|------|------|
| `projects` (`project_code LIKE 'PRJ-SEED-%'`) | 削除 → 再投入（このドキュメントの P-01〜P-13） |
| `projects` (`project_code LIKE 'PRJ-DEMO-%'`) | `DemoWorkloadSeeder` で残置 or 整理（衝突しない） |
| `approvals` (上記案件配下) | カスケード削除 → 再投入 |
| `tasks` (承認済案件配下) | 削除 → 再投入（T-01〜T-14） |
| `task_comments` / `task_histories` | タスク再投入に伴いクリア（必要に応じてサンプル投入） |
| `notifications` | 全削除 → 再投入（N-01〜N-09） |
| `users` / `departments` / `roles` / `permissions` | **触らない** |

## 6. 検証観点（投入後の確認）

| 観点 | 期待値 |
|------|--------|
| 案件一覧（申請タブ） | 5 状態がすべて 2 件以上表示される |
| 案件一覧（開発タブ） | `approved` 10 件すべてに進捗バーが描画される |
| 案件一覧（予算タブ） | 消費率 30% / 45% / 78% / 108% の 4 帯がカード色分けされる |
| 承認待ち一覧 | 部門承認待ち・本部承認待ちが各 2 件以上 |
| 再申請チェイン | `parent_project_id` の鎖が表示される（P-12 / P-13） |
| S-14 タスク一覧（カンバン） | 4 ステータス列すべてにタスクがある |
| S-14 タスク一覧（メンバー別） | 全ユーザー 10 名にタスクが表示される |
| 通知一覧 | 9 タイプすべてが少なくとも 1 件登場し、未読・既読が混在する |
| ヘッダー未読バッジ | 全ユーザーで未読件数 > 0 |
