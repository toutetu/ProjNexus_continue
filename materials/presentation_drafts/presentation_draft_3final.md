# ProjNexus プレゼンテーション最終構成

> 作成元: `materials/presentation_drafts/presentation_draft_2bone.md`  
> 位置づけ: PowerPoint 化の正本。過去ドラフトは材料庫として扱い、本資料では評価軸に沿って8枚本編に圧縮する。

---

## 全体方針

- 本編は **8枚**。課題1の工夫、課題2、感想を評価者が追いやすい順に並べる。
- 密度は課題PDFに近づけるが、1枚に詰め込みすぎず、表・図解・スクリーンショットで情報階層を作る。
- 22枚版ドラフトの詳細は、本文ではなく発表ノート・付録候補に回す。
- デザインはアプリのデザインシステムに合わせ、JPTレッド、ダーク、ブルー、シアン、イエローを使用する。

## スライド1: タイトル + アプリ概要

**主メッセージ**  
ProjNexus は、申請・開発管理・予算管理を一気通貫で扱う開発管理プロセス一元化アプリである。

**載せる要素**
- ProjNexus
- 開発管理プロセス一元化アプリケーション PoC
- Before: 申請システム / 部門別Excel / 予算管理Excel
- After: 申請・承認 / 開発管理 / 予算管理
- 提出者、期間、提出日
- スクリーンショット: `doc/manual/images/02_projects_approval_applicant.png`

**ノート**
ProjNexus は Project + Nexus。バラバラな管理をひとつの中心点に集約し、3ロールの視点をつなぐ意味を込めた。

## スライド2: 工夫1 設計重視

**主メッセージ**  
Phase 0 に全体の31%を投資し、設計の解像度を上げたことで、実装フェーズを短縮できた。

**載せる要素**
- Phase 別の見積 / 実績の比較
- 設計成果物: ER図、画面モック、AI.md、system_spec.md、components_spec.md、日次計画
- キーメッセージ: 設計の解像度が AI 時代の開発品質を決める

**ノート**
単にAIへ実装を任せたのではなく、AIが迷わないための前提・制約・正本資料を先に整えた点を強調する。

---

## スライド3: 工夫2 AIとの共同作業

**主メッセージ**  
AIを「自動生成ツール」ではなく、設計・実装・引継ぎを分担する協働相手として運用した。

**載せる要素**
- 自分: 判断、優先順位、最終責任
- Claude: 設計整理、仕様の言語化、比較検討
- Cursor: 実装、差分修正、テスト補助
- AI.md / system_spec.md / next_chat_handover による逸脱防止

**ノート**
AI利用そのものではなく、AIを使うための運用設計をした点が工夫。セッションをまたいでも判断がぶれないようにした。

---

## スライド4: 工夫3 日次ドキュメント & フェーズ管理

**主メッセージ**  
日次で設計・進捗・次回作業を更新し、やることだけでなく「やらないこと」も管理した。

**載せる要素**
- 3層ドキュメント: 設計の正本 / 日次ログ / 次チャット引継ぎ
- チェックリスト型フェーズ管理
- スコープ判断: 必須 / MVP / 後回し
- 効果: 迷いを減らす、手戻りを抑える、最終週に資料へ移れる

**ノート**
プレゼンでは「日報を書いた」ではなく、日報をスコープ管理の道具にしたことを伝える。

---

## スライド5: 工夫4 インタラクティブなUI

**主メッセージ**  
Backlog風のわかりやすさを取り入れつつ、PoCの業務に合わせた独自UIへ発展させた。

**載せる要素**
- 1画面3タブ: 申請 / 開発 / 予算
- ステータスバッジ、承認ステッパー、予算消費バー
- 4値運用: open / in_progress / resolved / closed
- スクリーンショット: `09_projects_dev.png`, `10_projects_budget.png`
- infotip で必要な情報を管理しつつ表示

**ノート**
Excel置き換えでは、見た目のわかりやすさが定着率に直結する。単なるCRUDではなく、現場が一覧で把握できるUIにした。

## スライド6: 課題2 実装した機能

**主メッセージ**  
課題2では、現場での運用性を高める機能を、実装コストと効果のバランスで選んだ。

**載せる要素**
- S-14 タスク一覧: カンバン / メンバー別 / 一覧の3ビュー
- 4値運用: 実装者完了報告 → 確認者OK
- タスク変更履歴の自動記録
- 通知拡張: 割当、期限、完了報告、確認OK

**ノート**
「全部実装する」のではなく、ヒアリングで効果が高いものから選んだ点を強調する。

## スライド7: 課題2 今後実装したいこと

**主メッセージ**  
実運用に進めるなら、管理範囲の拡張、予算管理の精緻化、外部出力を次に実装する。

**載せる要素**
- ロール別アカウント管理
- 予算実績の追加方式
- 予算アラート通知

## スライド8: 振り返り・感想

**主メッセージ**  
AI時代の開発では、実装力だけでなく、設計を言語化し、判断を残す力が重要だと学んだ。

**載せる要素**
- うまくいったこと: AIを使った設計、早い段階で画面イメージを作れたこと
- 想定外だったこと: モック作成のトークン消費、画面数とデザイン統一の調整
- 3つの学び: 設計の正本を作る、スコープを切る、AIとの引継ぎを設計する
- 実運用に進めるなら: ダッシュボード、権限管理、予算履歴

**ノート**
最後は技術説明ではなく、インターンを通じて得た開発プロセス上の学びとして締める。
