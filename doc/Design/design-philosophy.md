## 1. 「二重入力の解消」という一貫したテーマ

このアプリケーションの設計全体を貫くテーマは「二重入力の解消」である。

### 申請から予算管理まで一気通貫

申請 → 承認 → 進捗管理 → 予算管理がすべて同一アプリ内で繋がる。Excel → メール → 申請システムへの再投稿がすべて不要になる。

### 変更履歴の自動記録で報告作業を不要に

task_histories テーブルにより「誰が・いつ・何を変更したか」が自動記録される。これにより、本部への報告のための転記作業がなくなる。各部門のExcelを本部へメールで共有していた運用が、リアルタイムのデータ参照に置き換わる。

### 一元管理による効率化と可視化

この一貫性が、単なる「Excelの置き換え」ではなく「業務プロセスの改善」につながる

## 2. 将来の拡張を見据えたテーブル設計

### 基本方針

「PoCのテーブルは必要最小限に絞り、将来拡張は nullable FK カラムで備える」

設計を進める中で、将来のために空テーブルを先に作る方針から、nullable なFKカラムだけ残してテーブル自体は機能実装時に追加する方針に転換した。テーブルは独立して追加可能なため、既存テーブルに影響を与えずマイグレーション1本で足せる。一方、FKカラムは後から追加するとデータ移行が必要になるため、先に用意しておく価値がある。

### 切り分けの判断基準

| 判断 | 基準 | 例 |
|------|------|-----|
| PoCで実装 | 課題の必須要件に直結する | 承認フロー、タスクCRUD、予算管理、通知 |
| nullable FK のみ | 将来テーブルを参照する予定のカラム | tasks.parent_id、tasks.milestone_id |
| 将来テーブル追加 | 機能実装時にマイグレーションで追加 | milestones、project_members、添付ファイル等 |

### マイグレーション運用ルール（create と add の切替）

- **開発初期（ローカル単独で作業中）**  
  ER図の確定を優先し、`create_*` へ統合して読みやすさを重視してよい。`migrate:fresh` 前提で素早く整える。
- **共有開始後（他メンバーがpullした時点）**  
  既存 `create_*` は固定し、以降は `add_*` の追記だけで変更する。履歴の再現性と衝突回避を優先する。
- **切替の目安**  
  「ブランチを他者へレビュー依頼した時点」または「本番/検証環境へ初回デプロイした時点」で add-only へ移行。
- **禁止事項**  
  add-only移行後に、過去の `create_*` を書き換えて既存環境と差分を作らない。

### PoCの8テーブル

| テーブル | 理由 |
|---------|------|
| departments | ロール・権限の基盤 |
| users | 認証・ユーザー管理 |
| projects | 案件管理の中核（予算カラム含む） |
| approvals | 承認フローの監査証跡 |
| tasks | タスク管理の中核 |
| task_comments | Backlog風コメント機能 |
| task_histories | 変更履歴の自動記録 |
| notifications | アプリ内通知 |

### 将来追加するテーブル（機能実装時に作成）

| テーブル | 用途 | 既存テーブルへの影響 |
|---------|------|-----------------|
| milestones | マイルストーン・ガントチャート | tasks.milestone_id で参照 |
| project_members | 案件へのメンバーアサイン | project_id + user_id の中間テーブル |
| budget_items | 費目別予算管理 | project_id で参照 |
| task_attachments | タスクへのファイル添付 | task_id で参照 |
| project_comments | 案件レベルのコメント | project_id で参照 |
| project_attachments | 案件へのファイル添付 | project_id で参照 |

「時間がなかったから省いた」ではなく「意図的にスコープを定義し、拡張ポイントを明示した」という設計判断として説明できる。

## 3. 課題の理解

### 現状の問題

架空企業では、開発管理プロセスが「申請システム」「部門別Excel」「予算管理Excel」の3箇所に分散している。これにより以下の非効率が発生している：

- 各部門のExcelフォーマットが統一されておらず、本部でのデータ統合が手作業
- 進捗状況を申請システムへ再投稿する必要があり、二重入力が発生
- 予算消費率がリアルタイムで把握できない

### 私の洞察

部門ごとにExcelがバラバラなのは、各部門が管理したいポイント（粒度・観点）が異なるから。単に1つのフォーマットに押し込めるのではなく、統一されたデータ構造の上に柔軟な分類軸を設ける必要がある。

→ （課題2）Backlog風のタスク管理（種類・優先度・カテゴリ）を採用し、部門が自分に合った使い方を選択できるテーブル設計とした。将来的に拡張しやすい内容とした

## 4. 承認フロー：監査証跡を残す設計

### シンプルな一方向フロー

課題要件に従い、承認は一方向（申請者 → 部門管理者 → 本部管理者）とした。

```
draft → pending_dept → pending_hq → approved
                ↓              ↓
            rejected       rejected
```

却下された場合は申請者が作り直す。approvalsテーブルに承認/却下の履歴を残すことで、監査証跡を確保する。

### 独立した approvals テーブルを採用した理由

承認の記録方法として、projects テーブルにタイムスタンプ（dept_approved_at / hq_approved_at）を持たせる方式も検討した。しかしこの方式では却下時のコメント・承認者の記録場所がなく、再申請時に前回の承認記録が上書きされてしまう。

独立した approvals テーブルにより「誰が・いつ・どのレベルで・どんな判断をしたか」をレコード単位で追跡できる。却下の理由・経緯が記録されていると、再申請時の参考になり、将来的な監査対応にも有効である。


### 却下→再申請の設計：新規レコード + 元案件への紐づけ

却下→再申請の方式として、同一レコードの再利用ではなく、新規レコードにコピーして parent_project_id で元案件と紐づける方式を採用した。これにより却下時の申請内容・却下コメントが完全に保存される。再申請時は「この案件を元に再申請」操作で、元の案件情報をコピーした新しいプロジェクトが作成される。projects.revision カラムで何回目の申請かを即座に判定でき、parent_project_id のチェーンで申請の経緯も追跡できる。


### 承認ステップの可視化（課題2）

申請者から「申請した案件が今どの承認ステップにいるか分かると安心」というヒアリング結果があった。案件詳細画面にステータスバー（ステッパーUI）を配置し、現在の承認段階を一目で確認できるようにする。承認フェーズ中は全幅表示し、開発フェーズに移行後は「承認済みバッジ」に切り替えてクリックで履歴を展開する設計とした。

### 部門管理者が申請した場合のフロー

「申請者→部門管理者→本部管理者」のフローで、部門管理者が申請者になると自己承認が発生する。これは承認ワークフローの設計原則に反するため、部門管理者が申請した案件は部門承認をスキップし、本部承認のみのフローとした。

## 5. ロール設計

### 1ユーザーが複数ロールを持てる設計

現実の業務では「部門管理者が申請者を兼ねる」ケースが発生する。Spatieのlaravel-permissionを使い、1ユーザーに複数ロールを付与できる設計とした。ロール追加・変更がコード修正なしで対応できる。

### ロール別のデータアクセス

同じ「案件一覧」画面でも、ロールによって見える範囲を変える：

- **申請者** → 自分が申請した案件のみ
- **部門管理者** → 自部門の全案件
- **本部管理者** → 全部門の全案件

画面は1つだが、Controller側でクエリを分岐させることで、コードの重複を避けつつ適切なアクセス制御を実現する。

## 6. タスク管理：Backlog風の設計思想

### 課題の根本原因へのアプローチ

現状のExcel管理の問題は「フォーマットの不統一」だが、その根本原因は部門ごとに管理の観点が違うことにある。統一されたデータ構造（DB）の上に、柔軟な分類軸を設けることで解決する。

- `task_type`（種類）：バグ / 機能追加 / 改善 / タスク — 何をするか
- `priority`（優先度）：高 / 中 / 低 — どれだけ急ぐか
- `category`（カテゴリ）：設計 / 実装 / テスト 等 — どの工程か
- `status`（ステータス）：open → in_progress → resolved → closed — 今どこか

部門Aは「種類」で管理し、部門Bは「カテゴリ」で管理する、といった使い分けが可能。フォーマットは統一されているため、本部でのデータ集約は自動化できる。

### 4段階ステータスを採用した理由

3段階（todo / in_progress / done）と比較して、resolved → closed の段階を設けることで「担当者が完了報告 → 管理者が確認して閉じる」というレビューフローを表現できる。

### タスク担当の設計

PoCでは assignee_id による1タスク1担当者とし、実装負荷を抑える。タスク完了は担当者本人と同部門メンバーが操作可能とし、担当者不在時にも進捗が止まらないようにした。将来的には task_users 中間テーブルの導入で多対多担当に拡張できる。

### コメントと変更履歴

- **task_comments：** タスクごとの議論を記録。Excelではメールやチャットに散らばっていた情報をタスクに紐付けて一元化
- **task_histories：** 「誰が・いつ・何を変更したか」を自動記録。進捗の透明性を確保し、本部への報告を不要にする

### 将来拡張への対応

PoCではリスト表示＋詳細画面の実装とするが、DB設計は以下の将来機能に対応済み：

- **カンバンボード：** statusカラムでカラム分け
- **ガントチャート：** start_date / due_date + milestones で時系列表示
- **親子タスク：** parent_id でツリー構造を表現
- **フィルタ・ソート：** task_type / priority / category / assignee_id で柔軟な絞り込み
- **外部ツール連携：** 実運用時はBacklog・Jira・AsanaなどとのAPI連携を提案

## 7. 予算管理：PoCの最小要件に忠実に

### 課題の指示に従ったシンプルな設計

課題の指示「予算は案件単位の総額（予算額・実績額・消費率）を管理できれば可」に従い、projects テーブルに budget_amount（確定予算額）と actual_amount（実績額）を直接持たせる。消費率は actual_amount / budget_amount × 100 で算出する。費目別・月次管理は任意とされているため、PoCでは実装しない。

テーブル分離（budgets + budget_actuals）も検討したが、PoCのテーブル数を抑えてシンプルさを維持する方針を優先した。将来的に費目別管理が必要になった場合は budget_items テーブルを追加するだけで対応可能な設計としている。

### 予算と工数の管理方針

- タスクに「見積工数（人日）」「実績工数（人日）」を持たせ、完了時に入力
- 工数サマリーはタスクから自動集計
- 金額（外部委託費など）は案件単位で actual_amount に手入力
- 消費率は projects.actual_amount / projects.budget_amount × 100 で算出

内部開発は工数（人日）で管理するのが一般的であり、外部委託・ライセンス費などは工数と切り離して金額管理する方が現実的である。

### 開発管理フェーズへの移行と予算確定

projects.status が approved になった時点で、estimated_amount が budget_amount に転記され予算が確定する。以降、タスクの作成・進捗入力・予算実績の入力が解禁され、案件情報の編集はロックされる。

### 本部承認直後の初期タスク自動投入

承認後の最初のアクションを明確にするため、本部承認が完了した案件には初期タスク `実装計画作成` を自動投入する。見積工数は `3` 人日、担当者は申請者をプリセットする。これにより、承認直後の「最初に何をすればよいか分からない」状態を避け、開発フェーズへの遷移を滑らかにする。

### タスク通知の最小設計（課題1）

課題1では通知過多を避けながら、見落としが致命的になりやすいイベントに絞って通知する。

- `task_assigned`：タスク作成時・担当変更時に担当者へ通知
- `task_due_soon`：期限 3 日前 / 当日に担当者へ通知（日次ジョブ）
- `task_completed`：完了時に案件申請者へ通知

この3種を最小セットにすることで、実装コストを抑えつつ、運用上の「誰がやるか」「いつまでか」「終わったか」を確実に共有できる。

### 予算アラート

消費率70%超過で警告表示する設計とした。残り30%の時点でアラートを出すことで、対策を打つための時間的余裕が確保できる。閾値は将来的に設定値として変更可能にする。

## 8. 技術選定

### Laravel + Breeze (Inertia + React) + MySQL

| 選定 | 理由 |
|------|------|
| Laravel | 認証・ロール管理・通知・フォーム処理など、課題要件と得意分野が一致 |
| Breeze (Inertia + React) | SPAライクな体験をサーバーサイドルーティングで実現。REST API設計が不要でセッション管理がLaravel標準で完結 |
| MySQL | Laravelとの親和性が高く、リレーション設計がシンプルに表現できる |
| Tailwind CSS | Breezeに標準搭載。レスポンシブ対応（PC・タブレット）の要件を効率的に満たせる |
| Laravel Cloud | Laravel公式のクラウドサービス。デプロイが簡単で評価期間中の安定稼働を重視 |

### Inertia.js を選んだ理由

LaravelとReactを組み合わせる場合、通常はREST APIを別途設計する必要があり開発工数が増大する。Inertia.jsにより、LaravelのルーティングとReactのUI描画をシームレスに連携でき、APIの設計・実装コストを削減できる。また、Laravelの認証・ロール制御をそのままフロントに活かせるため、JWTトークン管理も不要になる。

## 9. 画面設計の思想

### ナビゲーション：業務フェーズによる分離

同一の案件が「申請される → 承認される → 開発が進む → 予算実績が入力される」というライフサイクルを持つことに着目し、ナビゲーションを **業務フェーズ軸** で3つに分離した。

| セクション | 業務フェーズ | 主な利用シーン | 主な利用者 |
|----------|-------------|-------------|----------|
| 申請・承認 | 起票〜承認完了まで | 案件を起案する / 承認待ちを裁く | 申請者・部門管理者・本部管理者 |
| 開発管理 | 承認済案件の進捗追跡 | 日次の進捗入力・タスク管理 | 申請者（主担当）・管理者 |
| 予算管理 | 承認済案件の予算消費追跡 | 月次の実績入力・消費率確認 | 申請者（主担当）・管理者（閲覧） |

#### なぜ「利用頻度」ではなく「業務フェーズ」で分けるか

機能軸（案件管理 / ダッシュボード / 承認 / 予算 / 通知）で分割すると、同じ案件の情報が複数画面に散らばり、ユーザーは「今どの画面を見ればいいか」を常に判断する必要が生じる。業務フェーズ軸で分離すると、**「今この案件で自分は何をすべきか」がナビゲーションから直接わかる** ため、ロール × フェーズで必要な画面に最短経路でたどり着ける。

### 案件一覧は1画面・フェーズ別タブで列を切替

3セクションはいずれも案件を起点にするため、**案件一覧は同一画面を使い回し、タブ切替で表示列を切り替える** 設計とした。

| タブ | 重視する列 | 意図 |
|------|----------|------|
| 申請タブ | ステータス・承認ステップ・申請日 | 自分の申請がどこで止まっているかを一覧で見たい |
| 開発タブ | 主担当・タスク進捗・期限 | 動いている案件の進み具合を見たい |
| 予算タブ | 予算額・実績額・消費率 | 予算消化の警戒ラインを俯瞰したい |

「画面数を最小化しつつ、役割ごとの情報密度を最大化する」という設計判断であり、実装コスト抑制（Controller 1本 + クエリ分岐）と UX の両立を狙う。

### ログイン直後のホームは暫定で「案件一覧（申請タブ）」

理想は本部管理者向けダッシュボードだが、ダッシュボードは課題2扱い（実装は後回し）としたため、PoCでは全ロール共通で「案件一覧（申請タブ）」をホームに据える。申請者は自分の申請状況、管理者は承認待ち案件への導線として機能する。

### 部門区別は文字列型で柔軟に

departments.type を string 型とし、現時点では department / headquarters の2値で運用する。boolean の is_headquarters と比べて、将来「子会社」「グループ会社」等の区分を追加する際にマイグレーション不要で対応できる。

## 10. 課題2（+α）の提案方針

### ダッシュボードによる可視化

本部管理者が「全案件の進捗を一覧で見たい」「予算消化率がリアルタイムで分かると早期に対策が打てる」と述べている。グラフ・チャートで全案件の進捗・予算状況を可視化するダッシュボードを実装する。

### レスポンシブ対応

申請者から「スマホからも更新したい」というヒアリング結果があり、課題要件でも「PC・タブレットいずれからでも全機能を利用可能」とされている。Tailwind CSSのレスポンシブユーティリティで対応する。

### 外部ツール連携の提案

PoCでは簡易タスク管理を内蔵するが、実運用時はBacklog・Jira・AsanaなどとのAPI連携を提案する。外部ツールとAPI連携することで、現場が使い慣れたUIを維持しつつ、申請・承認・予算管理の一元化が実現できる。「既存ツールを捨てない設計」は実際の導入障壁を下げる。

| 進捗確認の粒度 | 部門管理者=案件単位、申請者=タスク単位 | 画面設計に反映済み |
