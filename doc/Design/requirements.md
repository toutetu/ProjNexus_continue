# 課題要件整理 - 開発管理統合アプリケーション PoC

---

## 課題の全体像

架空企業の開発管理プロセスが「申請システム」「部門別Excel」「予算管理Excel」に分散しており、一元管理できるWebアプリのPoCを開発する。

---

## 課題1：必須要件

### 最終ゴール（課題資料より）

1. 案件の申請・承認ができる
2. 承認された案件の開発進捗を管理できる（※進捗管理方法は自身で検討）
3. 予算の消費状況を管理・確認できる（※最小要件：案件単位の総額で可。費目別・月次は任意）

### 承認フロー（課題資料の指示）

- 申請者 → 部門管理者 → 本部管理者の順で承認
- 承認は一方向でよい。却下された場合は申請者が作り直す
- 代理承認なども不要
- 通知はPoCではアプリ内通知のみで可（メール通知は任意）

### 承認フローの設計判断（自身で追加した仕様）

- 却下→再申請は新規レコードにコピー（parent_project_idで元案件に紐づけ）
- 部門管理者が申請した場合は部門承認スキップ（自己承認防止）
- approvalsテーブルで監査証跡を残す


### 承認後の開発管理フェーズ移行

projects.status が approved になった時点で以下が解禁される：

| 操作 | 承認前 | 承認後 |
|------|:---:|:---:|
| 案件情報の編集 | ○ | ✕（確定済み） |
| タスクの作成・編集 | ✕ | ○ |
| 初期タスクの自動作成 | ✕ | ○（`実装計画作成` / 3人日 / 担当=申請者） |
| 予算額の確定 | ✕ | ○（承認時に自動設定） |
| 予算実績の入力 | ✕ | ○ |
| 進捗率の更新 | ✕ | ○ |

### タスク通知（実装反映）

- タスク作成時、担当者へ通知（`task_assigned`）
- タスク担当変更時、新担当者へ通知（`task_assigned`）
- タスクが `resolved` になった時、確認者へ通知（`task_resolved`）
- 確認者が `closed` にした時、担当者・申請者へ通知（`task_reviewed`）
- タスクが `closed` になった時、案件申請者へ通知（`task_completed`）
- 期限通知は日次ジョブで実施（当日 / 3日前、`task_due_soon`）

### 現在管理している項目との対応

| 管理区分 | 管理項目 | 現在の管理場所 | PoCでの対応 |
|---------|---------|-------------|-----------|
| 申請情報 | 案件名 | 申請システム | projects.title |
| 申請情報 | 目的・概要 | 申請システム | projects.purpose |
| 申請情報 | 概算予算 | 申請システム | projects.estimated_amount |
| 申請情報 | 担当部門・申請者 | 申請システム | projects.department_id / applicant_id |
| 開発進捗 | タスク一覧・担当者 | 部門別Excel | tasks.title / assignee_id |
| 開発進捗 | ステータス・進捗率 | 部門別Excel | tasks.status / progress_rate |
| 開発進捗 | 開始日・期限 | 部門別Excel | tasks.start_date / due_date |
| 開発進捗 | 月ごとの進捗報告 | 部門別Excel | task_histories で自動記録（二重入力解消）。課題1で `TaskHistoryService` により status / progress_rate / assignee_id / due_date / priority を記録 |
| 予算管理 | 予算項目・予算額 | 予算管理Excel | projects.budget_amount |
| 予算管理 | 実績額・消費率 | 予算管理Excel | projects.actual_amount（消費率は算出） |

### 課題2の実装優先度

| 優先度 | 機能 | 理由 |
|--------|------|------|
| **実装**| 承認ステップの可視化（ステッパーUI） | 実装コストが低く、UX向上効果が大きい |

| **実装** | **`task_histories` 自動記録** | Excel の二重入力を解消する目玉機能。設計済みテーブルへの書込実装 |

| 低 | 予算アラート（70%超過で通知） | ダッシュボード上の表示で代替可能 |

## 考慮事項（課題資料より）

- 部門数はPoCでは2〜3程度でよい
- 本部はPoCでは1つでよい
- ロールごとのデータアクセス範囲は自身で検討の上、定義・実装
- 認証はアプリ独自のID/パスワード等の簡易方式で可（EntraID/SSOは実装対象外）
- PC・タブレットいずれからでも全機能を利用可能にしたい
