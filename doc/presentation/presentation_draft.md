## スライド 3: 課題理解 — 3ツールに分断された現場と、3ロールの声

> タグ: **[課題1 工夫]**

### このスライドで言いたいこと（1行）
開発管理プロセスは3つのツールに分断され、3ロールそれぞれに異なる悩みを抱えていた。

### 本文

#### Before — 既存環境の課題

開発管理プロセスは **3つのツールに分断** されており、情報が繋がらない状態だった。

| ツール | 用途 | 課題 |
|---|---|---|
| 専用申請システム | 案件の申請・承認 | 承認状況を確認するには専用画面へログインが必要 |
| **部門別 Excel** | タスク進捗管理 | **部署ごとにフォーマットがバラバラ**、横断把握できない |
| 予算管理 Excel | 予算・実績管理 | 月次集計が手作業、リアルタイム性なし |

→ **承認状況の確認に複数ツールを横断**し、誰が何を承認待ちなのかが把握しにくい。

#### ヒアリング — 3ロールの声

| ロール | 代表的な声 |
|---|---|
| **本部管理者** | 「全案件の進捗を一覧で見たい。**ダッシュボードが欲しい**」「予算消化率がリアルタイムで分かると早期に対策が打てる」 |
| **部門管理者** | 「Excelを本部に送る手間がなくなるのは嬉しい。**入力の手間は最小限に**」「部門内のメンバーのタスク状況を**一目で把握**したい」 |
| **申請者** | 「申請した案件が**今どの承認ステップにいるか**分かると安心」「進捗の入力をもっと簡単にしてほしい」 |

## スライド 4: 仮説と解決アプローチ — 多様な進捗管理機能で受け皿になる

> タグ: **[課題1 工夫]**

### このスライドで言いたいこと（1行）
3ロールの異なる声から「機能を盛り込む」仮説を立て、3本柱の一気通貫アプリで答えた。

### 本文

#### 立てた仮説

> **各部署が異なる形式の Excel で進捗管理している現状をひとつのアプリに移行するには、**
> **各部署のニーズを吸収できる多様な進捗管理機能を盛り込む必要がある。**

##### 仮説の根拠
- 部署ごとに Excel フォーマットが違う ＝ それぞれが必要とする「見え方」「機能」が異なる
- 単機能の置き換えでは、結局 Excel に戻ってしまう
- 多様な使い方を吸収できなければ、一元化のメリットが消える

#### 解決アプローチ — After

| 仮説への答え | 実装した機能 |
|---|---|
| ロール別に「自分が今やるべきこと」を即座に把握 | サイドバー3セクション × ロール別の初期画面 |
| 部門ごとの好みの見え方に対応 | カンバン⇄メンバー別マトリクスの**ビュートグル**（S-14） |
| Excel への転記作業を不要に | タスク変更履歴の**自動記録**（`task_histories`）|
| 部門境界を越えた情報共有 | 通知（担当・期限・完了・確認依頼）/ ロール別アクセス制御 |
| 品質ゲートの統一 | タスク **4値運用**（実装者の完了報告 → 確認者OK）|
| 早期に異常を察知 | 予算消費率の**警告色**（70%・100% 超） |

[画像挿入] `doc/manual/images/09_projects_dev.png`
キャプション: After — 案件一覧（開発タブ）。タブ切替で同じデータを「申請」「開発」「予算」3つの視点で見られる

### 補足コメント

このスライドは **工夫点③（一元化への仮説）の章への布石** にもなる。
仮説スタイルの記述は箇条書きより枠線で囲んだ「主張ボックス」で見せると、視線が止まる。
After の表は左列を「仮説（声）」、右列を「実装」にすると、ヒアリングと実装の対応が一目で分かる。

## スライド 5: 工夫② AI協働 — 「判断は自分・実装は AI」の役割分担

> タグ: **[課題1 工夫]**

### このスライドで言いたいこと（1行）
AI を「丸投げ先」ではなく「判断のスパーリングパートナー」として使った。

### 本文

#### 採用した AI ツール

| AI ツール | 主な用途 |
|---|---|
| **Claude**（Anthropic） | 設計相談・モック作成（HTML）・ドキュメント整合性チェック・コードレビュー |
| **Cursor**（実装エージェント） | コード生成・migration 作成・テスト雛形 |

#### 役割分担（重要）

| 担当 | 役割 |
|---|---|
| **自分** | ヒアリング解釈・スコープ判断・**最終承認**・要件文書化 |
| Claude | 設計案の比較・モック生成・整合性レビュー |
| Cursor | 設計に従った実装・テスト |

→ **AI に判断させない**。判断責任を自分が持ち、AI は実装スピードを倍化する道具として使う。

#### AI 向け指示書 `AI.md`（自前で整備）

- 設計思想・スコープ・実装ルールを **約 360 行** で明文化
- 「課題1 / 課題2 のスコープ」「実装前のプラン承認義務」「コーディングルール」を記述
- AI が自律的に逸脱しないよう、**明確な制約を与える** ことで一貫性を担保

[画像挿入] なし（コンセプト図 or 役割分担表のみ）

### 補足コメント

AI を使うこと自体は珍しくない。差別化ポイントは「AI に判断責任を委ねないルール作り」。

## スライド 8: 工夫② フェーズ分けチェックリスト — スケジュールを安全に進める

> タグ: **[課題1 工夫]**

### このスライドで言いたいこと（1行）
5フェーズに分け、各フェーズをチェックリストで管理することで、設計と実装の取りこぼしをゼロにした。

### 本文

#### Phase 別の構造とチェックリスト

| Phase | 内容 | 見積 | 実績 | チェック項目数 |
|---|---|---|---|---|
| 0 | 設計・環境構築 | 35h | **31h** ✅ | 14項目 |
| 1 | 認証・レイアウト・共通UI | 8h | **9h** ✅ | 14項目 |
| 2 | 申請・承認フロー（核） | 20h | **10h** ✅ | 17項目 |
| 3 | 開発・予算管理 MVP | 10h | **6h** ✅ | 22項目 |
| 4 | +α（4値運用・S-14・履歴） | 5h→拡大 | 進行中 | 16項目 |
| 5 | ドキュメント・プレゼン・最終確認 | 20h | 予定 | 14項目 |

各 Phase の項目は `intern_schedule.md` で `- [x]` / `- [ ]` 形式の Markdown チェックリストで管理。

#### チェックリスト管理の効果

| 効果 | 内容 |
|---|---|
| **取りこぼし防止** | 各 Phase の完了条件が明文化されているため、漏れに気付かないまま終わらせるリスクをゼロに |
| **進捗の可視化** | 「Phase 3 は 22 項目中 17 項目 完了」のような状態を即座に把握 |
| **スコープ判断の補助** | 「実装 (✅)」「設計のみ (📐)」をチェック状態で区別 |
| **AI への伝達** | チェックリストをそのまま AI に渡せば、進捗状況を即理解してもらえる |

#### 3層の優先順位（スコープ判断のフレームワーク）

| 層 | 区分 | チェック対象 | 評価への効果 |
|---|---|---|---|
| ① 必達 | 課題1 必須 | 完了が前提 | **完成度** で評価 |
| ② +α | 課題2 として実装 | 実績時間と相談しながらチェック | **発展力** で評価 |
| ③ 設計のみ | 拡張構想 | チェック対象外（プレゼン素材として残置）| **構想力** で評価 |

#### スコープ判断の問いかけ

- 「これなくして提出物として成立するか？」 → No なら ① 必達
- 「これでアピール度が大きく上がるか？」 → Yes なら ② +α
- 「これがないと弱みに見えるか？」 → ③ 設計のみで補強

「やらなかった」ことを正面から扱うのが肝心。スコープ管理は実務エンジニアの重要能力。

## スライド 9: 工夫③ — 一元化への仮説：「機能を盛り込む」必要性

> タグ: **[課題1 工夫]**

### 本文

#### 立てた仮説（ProjNexus の設計の出発点）

> **各部署が異なる形式の Excel で進捗管理している現状を、ひとつのアプリに移行するには、**
> **各部署のニーズを吸収できる「多様な進捗管理機能」を盛り込む必要がある。**

#### 仮説の根拠

- 各部署の Excel フォーマットが違う ＝ それぞれが必要とする「見え方」「運用ルール」が異なる
- **単機能の置き換えでは結局 Excel に戻る** — 一元化が機能不足で破綻するリスクが大
- 多様な使い方を吸収できなければ、一元化メリットそのものが消える
- 「Excel じゃなきゃダメ」と言われない柔軟性が、移行成功の最低条件

#### 仮説のメッセージ

「**最小実装で済ませる**」と「**多様性を吸収する**」は背反する。
PoC でも前者に傾きがちだが、本プロジェクトでは **後者** を選んだ。
それは、**この PoC が実運用に進む可能性** を本気で考えた結果である。

## スライド 11: 工夫③ — 仮説検証の難しさと、観察可能な設計

> タグ: **[課題1 工夫]**

### このスライドで言いたいこと（1行）
PoC段階で仮説検証はできない。だからこそ、実運用時に検証できる設計を残した。

### 本文

#### 直面した課題

- PoC では実ユーザーがいない → **使用ログが取れない**
- 仮説の根拠は **ヒアリング3声**（定性データ）のみ
- 「6つの機能が本当に必要だったか」を**定量検証する手段がない**

#### 解決アプローチ — 観察可能な設計を残置

仮説検証は「実運用に進んだとき」に行えるよう、**観察可能性** を機能の中に組み込んだ：

| 観察手段 | 検証できる仮説 |
|---|---|
| `task_histories` の蓄積パターン | どの操作が頻繁か → 機能の利用度合い |
| 通知の既読率（読まれた通知の割合）| 通知機能の到達度・有効性 |
| メンバー別マトリクスのアクセス頻度（部門管理者）| 「俯瞰ニーズ」仮説の真偽 |
| カンバン⇄マトリクスの切替比率 | 「部署ごとの好みの見え方」仮説の真偽 |
| 4値運用での `resolved → closed` 滞留時間 | 確認工程が機能しているかの実態 |

→ いずれも**既存のテーブル設計で取得可能**（追加 migration 不要）。

#### 学び

- **仮説駆動開発** = 仮説を立てて作るだけでなく、**仮説を見直す仕組み** を組み込むこと
- 「実装したら終わり」ではなく「**観察できる仕組み**」を残すことが、データ駆動の改善に繋がる
- PoC の限界を正直に認識することが、プロダクトの次の成長を可能にする

## スライド 16: 実装ハイライト② — 同一画面・3タブで列を切り替える設計

| タブ | 重視する列 | 想定ユーザー |
|---|---|---|

| **タスク変更履歴** | Excel への転記・本部への報告メール | `TaskHistoryService` が **status / progress_rate / assignee_id / due_date / priority** の5項目を自動記録 |
| **初期タスク投入** | 承認直後の「最初に何をするか」の判断 | `ApprovalService::approveHq` 内で「実装計画作成」（3人日・担当者=申請者）を自動 INSERT |
| **通知5種** | 関係者への都度連絡 | `task_assigned` / `task_due_soon`（日次ジョブ）/ `task_completed` / `task_resolved` / `task_reviewed` |
| **消費率の警告色** | Excel の手動色付け | `actual_amount / budget_amount × 100` を **70% / 100% 超で警告色**（DBには持たず算出） |

#### 設計のポリシー（自動化の「節度」）

- **通知は最小セットで過多を避ける** — 「誰がやるか / いつまでか / 終わったか / 確認できたか」に絞る（ノイズ通知は無価値）
- **算出値は DB に持たない** — 消費率や案件進捗率はリアルタイム算出で整合性を担保（更新漏れによるズレを構造で排除）
- **履歴は表示用文字列で保存** — UI そのままを `old_value` / `new_value` に格納し、Enum 変換ロジックを再表示時に持ち込まない

「自動化」は派手な機能ではないが、**Excel 文化からの脱却を一番強く支える基盤**。

## スライド 19: 課題2 ② — タスク4値運用（実装者→確認者の品質ゲート）

> タグ: **[課題2]**

### このスライドで言いたいこと（1行）
3値（todo/in_progress/done）では表現できない「完了報告 → 確認OK」の工程を、4値運用と確認者カラムで構造化した。

### 本文

#### なぜ3値では不足か

| 状況 | 3値での表現 | 課題 |
|---|---|---|
| 担当者「実装終わったので確認お願いします」 | done に倒すしかない | 確認前なのに完了扱い／差し戻し時の状態が曖昧 |
| 確認者「OK、閉じます」 | 既に done なので操作なし | 「誰が品質を保証したか」が記録されない |

#### 4値運用の設計

```
open → in_progress → resolved → closed
            ↑              ↑
       担当者が遷移      確認者が遷移
```

- **`reviewer_id` カラム追加**（nullable FK→users）で確認者を明示的に保持
- **遷移権限を分離**:
  - `in_progress → resolved` は **担当者（assignee_id）のみ**
  - `resolved → closed` は **確認者（reviewer_id）のみ**
- **Policy + Service 二層** で担保（`ProjectWorkItemPolicy` + `ProjectTaskController`）

#### 通知の追加（最小セットを維持）

| 通知 | タイミング | 受信者 |
|---|---|---|
| `task_resolved`（確認依頼）| 担当者が `resolved` に遷移 | 確認者 |
| `task_reviewed`（確認OK）| 確認者が `closed` に遷移 | 担当者 |

→ 既存3種（assigned/due_soon/completed）と合わせて **5種で「誰がやるか／いつまでか／終わったか／確認できたか」を完全カバー**。

#### 仮説への対応

> 仮説:「部署ごとの品質ゲートを統一する」
> → 実装者と確認者を **構造的に分離** することで、部署ごとの非公式な確認運用をアプリ上に乗せた。

キャプション: タスク詳細 — 担当者・確認者を明示し、ステータス遷移ボタンを権限別に表示

### 補足コメント

PowerPoint では中央に状態遷移図を大きく置き、**遷移ごとに「実行できる人」を色分け**で示すと一目で伝わる。

## スライド 22: 拡張構想② — 予算実績の追加方式 + 予算アラート

> タグ: **[課題2]**

### このスライドで言いたいこと（1行）
PoC では「上書き方式」で実装した予算管理を、**実運用に進めた瞬間に「追加方式」へ移行する設計** を残した。

### 本文

#### 上書き方式 → 追加方式への移行構想

| 区分 | 課題1（PoC で実装）| 課題2（拡張構想）|
|---|---|---|
| 実績の保持方法 | `projects.actual_amount` を**上書き更新** | **`budget_actuals` テーブル**へ INSERT（1支出=1行）|
| `actual_amount` の扱い | 唯一の実績値 | `SUM()` の集計キャッシュ |

| 移行手順 | — | `budget_actuals` 追加 + 既存値の初期 INSERT + Service 切替 |

#### 予算アラート通知（既存通知5種への追加）

| 閾値 | 通知タイプ | 受信者 |
|---|---|---|
| 86% 超過 | `budget_alert_warning` | 主担当・部門管理者 |
| 100% 超過 | `budget_alert_exceeded` | 主担当・部門管理者・本部管理者 |

→ 既存の `NotificationType` Enum + `NotificationService` に **2件追加するだけ** で運用可能（設計済み）。

#### 実装しなかった理由

- 課題1 では「上書き方式」が要件として明記されている（`AI.md §1`「★ 案件単位の予算・実績管理（上書き方式）」）
- 監査証跡が必要になるのは **複数人で予算を入力する実運用フェーズ** であり、PoC のシナリオでは過剰機能
- 移行手順を明確に残しておけば、PoC → 実運用の橋渡しは1スプリントで可能

#### 個人的な感想

- 課題を「完成させる」だけでなく、**「実運用に進めたら何が残るか」** を考え抜くことが、PoC の本当の価値だと学んだ
