# 要件定義書（設計確定版ドラフト）

## 1. 文書情報
- 文書名: quest_1 要件定義書
- 版数: v0.2
- 更新日: 2026-04-13
- 作成者: インターン開発チーム
- 対象プロジェクト: 社内案件・タスク・予算管理アプリケーション

## 2. 背景・目的
インターンシップ開発において、案件申請から承認、開発進行、予算管理までを一元管理する。
本書は、実装・レビュー・受け入れの基準を統一し、開発を円滑に進めることを目的とする。
また、課題文で指定された提出要件（課題1・課題2・提出物）を満たすことを目的とする。

## 3. 技術スタック
- バックエンド: Laravel 11
- フロントエンド: React 18 + TypeScript
- SPA連携: Inertia.js
- UI: Tailwind CSS / shadcn/ui
- DB: MySQL 8
- デプロイ先: Laravel Cloud

## 4. スコープ
### 4.1 対象範囲（In Scope）
- 3ロールに対応した案件・タスク・予算管理機能
- 承認フロー（申請者 -> 部門管理者 -> 本部管理者）の実装
- タスク実績工数に基づく予算集計表示
- 11画面の実装と主要6画面の操作性担保
- 課題1（必須機能）と課題2（改善提案・追加実装）の実施

### 4.2 対象外（Out of Scope）
- 外部タスク管理ツールとのリアルタイム連携
- 本番運用を前提とした高負荷試験
- 高度な分析ダッシュボード（予測分析・AI分析）
- EntraID/SSO連携（背景情報でありPoC実装対象外）

## 5. ロール・権限要件
### 5.1 ロール定義
- 申請者
- 部門管理者
- 本部管理者

### 5.2 ロール仕様
- 1ユーザーは複数ロールを保有可能とする。
- ログイン時または画面上で、利用ロールを切り替えて表示・操作を行えること。
- 本PoCでは部門数は2〜3、本部は1つを想定する。

## 6. 業務フロー要件
### FR-01 案件申請・承認フロー
- 申請者は新規案件を申請できること。
- 承認は「申請者 -> 部門管理者 -> 本部管理者」の一方向フローであること。
- 各承認状態（未申請、申請中、部門承認済み、本部承認済み、差戻し等）を追跡できること。

### FR-02 案件フェーズ管理
- 案件詳細画面で承認フェーズと開発フェーズを切り替えて表示できること。
- フェーズに応じて表示項目・操作可能項目を制御できること。

### FR-03 タスク管理
- 案件に紐づくタスクを作成、更新、完了にできること。
- タスクに予定工数・実績工数を保持できること。
- タスク完了承認は担当者本人または同部門メンバーが実施可能であること。

### FR-04 予算管理
- 工数はタスク実績から自動集計すること。
- 金額は案件単位で別途入力できること。
- 本部管理者向けに棒グラフと詳細一覧を表示できること。
- PoC最小要件として、案件単位の総額（予算額・実績額・消費率）を管理できること。

### FR-05 通知
- 承認依頼、承認結果、差戻し、タスク更新等のイベントに対する通知を管理できること。

## 7. 画面要件
### 7.1 画面数
- 全11画面を実装対象とする。

### 7.2 最終確定版 画面一覧
1) ログイン（全員）
- ID/パスワード認証

2) ダッシュボード（全員）
- ロール別ウィジェット切り替え

3) 新規案件申請フォーム（申請者・部門管理者）
- 3ステップ入力
- 部門管理者で申請した場合は本部承認のみのフロー

4) 案件一覧（全員）
- 申請者: 自分の案件のみ表示
- 部門管理者: 自部門の案件を表示
- 本部管理者: 全部門の案件を表示

5) 案件詳細（全員）
- 承認フェーズ中は承認フローバーを表示
- 開発フェーズ中は承認フローバーを非表示

6) タスク一覧（申請者）（申請者）
- 自分担当の全案件タスクを横断表示
- フィルター・検索機能を提供

7) タスク一覧（部門管理者）（部門管理者）
- 全タスク一覧ビューと案件別グループビューを切り替え

8) 予算管理（本部管理者）
- 部門別・案件別の棒グラフ＋詳細一覧を表示
- 工数サマリーを表示

9) 通知一覧（全員）
- タイプ別カラーボーダー表示
- フィルタータブを提供

10) 承認・却下モーダル（部門管理者・本部管理者）
- 案件詳細・ダッシュボードから呼び出し
- 却下理由の入力を必須とする

11) ユーザー管理（本部管理者）
- テストアカウント作成
- ロール割り当て
- PoC必須機能として扱う

## 8. データ要件（ER設計）
### 8.1 テーブル構成（8テーブル）
- users
- departments
- projects
- tasks
- task_users
- budgets
- budget_actuals
- notifications

### 8.2 主要データ仕様
- `tasks` テーブルに `planned_effort`（予定工数）、`actual_effort`（実績工数）を保持すること。
- リレーションは案件中心（projects）で整合を取ること。
- 監査用に作成日時・更新日時を保持すること。

## 9. 非機能要件
### NFR-01 性能
- 一覧表示、承認操作、タスク更新など主要操作は通常条件で3秒以内を目標とする。

### NFR-02 セキュリティ
- ロールに応じた認可制御を行うこと。
- 入力値バリデーションを実施し、不正入力を保存しないこと。
- 画面・APIエラー時に内部情報を露出しないこと。

### NFR-03 保守性
- フロントエンド/バックエンドで責務分離し、命名規則を統一すること。
- 将来的な外部ツール連携を考慮し、タスク機能は疎結合に設計すること。

### NFR-04 可用性
- ローカル開発環境および検証環境で安定動作すること。
- 対応ブラウザは Microsoft Edge を優先対象とする。

### NFR-05 利用端末
- PC・タブレットの双方で主要機能を利用できるUIとする。

## 10. 課題別要件
### 10.1 課題1（必須）
- 案件の申請・承認ができること。
- 承認された案件の開発進捗を管理できること。
- 予算の消費状況を管理・確認できること。

### 10.2 課題2（＋α）
- 実運用での使いやすさ向上に向けたルール整備または機能追加を提案すること。
- 可能な範囲で提案内容を実装すること。
- 例として、全案件の進捗・予算状況のダッシュボード可視化を含めてもよい。
## 11. 制約条件
- 開発期間: インターンシップ期間内
- 開発方針: タスク管理は簡易実装を採用（将来的に外部ツール連携を提案）
- インフラ: Laravel Cloudへのデプロイを前提とする
- 認証はPoCとして簡易なID/パスワード方式で可とする
## 12. 受け入れ条件
- 3ロールで画面表示と操作権限が正しく切り替わること。
- 承認フローが一方向で正しく遷移すること。
- タスクの予定工数・実績工数が保存され、予算画面に反映されること。
- 最終確定した11画面の主要導線が仕様どおりに動作すること。
- 本部管理者画面で予算の棒グラフと詳細一覧が表示されること。
## 13. 提出物要件
- `main` ブランチを評価対象として、Webアプリ本体を提出すること。
- `main/doc/Information.md` にデプロイURLと動作確認用ID/Passを記載すること。
- ER図、利用マニュアル、プレゼン資料（`presentation_{name}.pdf` または `.md`）を `main/doc` 配下に保存すること。
- 提出後に評価者が動作確認できる状態を維持すること。
