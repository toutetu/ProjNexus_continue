# S-05 新規案件申請 方針


## 構成

- **背景**: 申請者・部門管理者が新しい開発案件を起案するフォーム画面。サイドバー「新規申請」またはS-03aの「新規申請」ボタンから遷移
- **レイアウト**: AuthenticatedLayout（サイドバー「申請・承認 > 新規申請」がアクティブ）、max-w-3xl（フォーム画面なので一覧より狭め）
- **パンくず**: 申請・承認 > 新規申請
- **URL**: `/projects/create`
- **アクセス**: applicant と dept_manager のみ。hq_manager は申請不可（§6 ロール別アクセス制御）

## 構成セクション（上から下へ）

### ページヘッダー
- 「新規案件申請」タイトル + サブテキスト「開発案件を起案し、部門管理者・本部管理者の承認を受けます」
- 右端に「一覧に戻る」リンク（← arrow-left アイコン + テキスト）— muted色、hover でdarkに
- ※ S-03a / S-04 のような `.title-accent` 縦バーは **付けていない**（フォーム画面なので装飾を抑制）

### 承認フローガイド（ApprovalFlowGuide）
- グラデーション背景のバナー（`linear-gradient(135deg, rgba(cyan,0.08), rgba(purple,0.06))`）
- 4ステップを横並び: ①あなたが申請（dark、太字）→ ②部門管理者が承認（cyan）→ ③本部管理者が承認（blue）→ ④承認・開発フェーズへ（green ✓）
- **dept_manager が自身で申請する場合**: ②をスキップ表示にする（`skipDept: true`）— 直接 pending_hq へ遷移するため
- 初回申請者が「申請後に何が起きるか」を理解するための導線。2回目以降は折り畳みも可

### フォームカード（白背景、border + shadow-sm）
- ヘッダー: 「案件情報」+ file-plus-2 アイコン（jpt-blue）

#### フィールド構成（上から順に）

1. **案件名** *（必須）
   - テキスト入力、maxlength=80、placeholder「例：次世代EAMシステム開発」
   - 右下にリアルタイム文字数カウンター（`0 / 80`、mono フォント）
   - ヘルプ: 「一覧で識別しやすい名称を推奨（80文字まで）」

2. **担当部門** *（必須）
   - セレクトボックス: 開発1部 / 開発2部 / 開発3部
   - ヘルプ: 「案件の主担当となる部門を選択。部門管理者がこの部門の承認者として割り当てられます」
   - React実装では `usePage().props.departments` から選択肢を動的に生成

3. **目的** *（必須）
   - テキストエリア、rows=4
   - ヘルプ: 「承認者が投資判断をする際の最重要項目。定量的な目標があれば含めてください」

4. **概要・説明**（任意）
   - テキストエリア、rows=7
   - ヘルプ: 「任意。承認後は編集ロックされるため、可能な範囲で詳細に」

5. **概算予算** *（必須） + **概算工数**（任意） — grid-cols-2 横並び
   - 予算: AmountInput（右寄せmono + 自動カンマ区切り + 右端に「円」単位）
   - 工数: AmountInput（右端に「人日」単位）
   - 予算ヘルプ: 「承認時に『確定予算』として確定し、以降は変更不可」
   - 工数ヘルプ: 「任意。タスクの合計見積と比較する参考値」

6. **下書き復元ヒント**（条件付き表示）
   - アクセントカラーの破線ボーダー + 薄黄背景（`#FFF9E6`）
   - 💡アイコン + 「下書きから復元できます」+ 下書き件数 + リンク
   - 過去に draft 保存した案件が存在する場合のみ表示

### フォームフッター（bg-gray-50、border-top）
- 左: 「下書き保存」ボタン（btn-outline、save アイコン）— status=draft でDBに保存、一覧に戻る
- 右: 「キャンセル」テキストボタン + 「申請する」ボタン（btn-primary、send アイコン）
- 「申請する」クリック → 確認モーダル（ConfirmDialog）を開く

### フォーム下注釈
- info アイコン + 「申請後は編集できません。承認・却下の結果は通知で届きます。却下された場合は『再申請』から内容を引き継いで作成できます。」
- muted色 text-xs で控えめに

## 内蔵モーダル: 申請確認（ConfirmDialog）

- max-w-[480px]、backdrop blur、Esc/背景クリックで閉じる
- ヘッダー: send アイコン（cyan背景の丸）+ 「申請しますか？」
- 説明文:「申請後は内容を編集できません。承認・却下を待つステータスになります。」
- グレーカードにサマリ:
  - ステータス: `pending_dept` の StatusPill + 「へ」
  - 次の承認者: 「{部門名} 部門管理者」
  - 通知:「承認/却下時にメール + アプリ内通知」
- フッター: キャンセル + 「申請する」（btn-primary）
- dept_manager が自身で申請する場合: ステータスは `pending_hq` を表示し、次の承認者は「本部管理者」

## 設計上のこだわり

- **「下書き保存」と「申請」を明確に分離** — 「下書き保存」は左にアウトラインボタン、「申請する」は右にプライマリボタン。誤って申請してしまう事故を防ぐ。下書きは draft ステータスで DB に保存され、一覧の申請タブに表示される
- **確認モーダルで「何が起きるか」を事前提示** — ステータス遷移先・次の承認者・通知方法を灰色カードで要約。S-04 の承認モーダルと同じ「アクション前の確認」パターンを踏襲
- **フローガイドは「初回ユーザーの不安を解消する」ため** — 承認フローの全体像を申請前に見せることで、「申請したら次に誰が何をするのか」を理解してから入力に進める。S-04 の ApprovalStepperFull が「今どこにいるか」なら、こちらは「これからどう進むか」
- **ヘルプテキストは承認者目線で書く** — 目的の「承認者が投資判断をする際の最重要項目」のように、入力者に「何のためにこの情報が要るのか」を伝える。単なる「入力してください」ではモチベーションが湧かない
- **max-w-3xl の狭幅** — 一覧（max-w-7xl）や詳細（max-w-7xl）より狭いのはフォームの可読性のため。テキストエリアが横に広がりすぎると読みにくい
- **フォーカスリングは jpt-blue** — 入力中のフィールドが視覚的に明確。全フォーム要素で統一（border-color + box-shadow の2層）
- **概算予算の自動カンマ区切り** — AmountInput コンポーネントで内部的に文字列⇔数値を変換。表示は `10,000,000` だが onSubmit で送る値は `10000000`（number型）
- **「承認後は編集ロック」の明示が複数箇所** — 説明フィールドのヘルプ + フォーム下注釈の2か所で伝達。「あとで直せると思って雑に書く」ことを防ぐ
